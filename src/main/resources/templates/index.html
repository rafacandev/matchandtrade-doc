Welcome to Match and Trade

Match and Trade is a web application where many people trade several items at once using an algorithm to determine which items will be traded. A typical trade normally follow distinct steps:
  • An organizer creates a trade
  • Members subscribe to the trade and submit their items
  • Members choose which items they would like to trade
  • The organizer closes the trade and the website generates a list with the items to be traded
  • Members exchange items according to the generated list

Endpoints vs Resources

Generally speaking, resources have consistent format and behavior whereas endpoints may vary in format and behavior. Resource URLs are a specialized endpoints which provides standard operations over a resource. Example: resources support HATEOAS, pagination and query parameters are relevant to the given resource. Endpoint URLs may not follow all constraints of resource URLs. Some may also perform actions (e.g.: authenticate). See Web Services - Terminology for more details.

HATEOAS

The importance of HATEOAS cannot be emphasized enough. This application uses the Spring HATEOAS approach to handle hypermedia

Expanding

Expanding resources (see Atlassian API for an example) is discouraged while multiple asynchronous calls are favored.

Pagination

REST clients should rely on the pagination information which is included in responses with multiple results. The pagination follows the LinkHeader specification along with RFC5988. It works similarly to GitHub's pagination.

When performing a GET to resources that returns an array you can pass the query parameters _pageSize to indicate the number of records returned in a page and _pageNumber to indicate which page number you want to return. Note that _pageNumber start at number 1.

Example:

${REST_GUIDE_PAGINATION}

PUT vs PATCH

PUT requests are favored over PATH requests. While PATH may offer smaller payloads, it also introduces development complexity for little benefit.

Additionally, when doing a PUT or PATCH it is recommended to do not include the id for the given resource in the payload. For example, when doing a PUT /trades/{tradeId} do not include tradeId in the payload, it will be simply ignored.

Example:

----- Request ----- PUT http://localhost:8081/rest/v1/trades/1 Headers: {Authorization: 7075-1255-9178}{Content-Type: application/json} { // Do not enter tradeId in this payload, it will be ignored. "name" : "Name to update" } ----- Response ----- Status: HTTP/1.1 200 { "name" : "Name to update", "tradeId" : 1, "_links" : [ { "rel" : "self", "href" : "http://localhost:8081/rest/v1/trades/1" } ] }
